🐚 Shell笔记：入门与学习变量

一、Bash 脚本的四种执行方式

在实际开发中，执行 Shell 脚本有多种途径，根据你的需求（是否隔离环境、是否需要执行权限）可以选择不同的方式：

🔹 方式 1：使用 bash / sh 解释器执行

✅ 优点：无需脚本具备执行权限（+x）
⚠️ 注意：会在子 Shell 中运行

🔹 方式 2：直接路径执行（推荐日常使用）

✅ 前提：需提前赋予执行权限 chmod +x hello.sh
⚠️ 注意：同样在子 Shell 中运行

🔹 方式 3：使用 source 命令加载

✅ 在当前 Shell 环境中执行，变量/函数变更会直接影响当前终端会话
✅ 非常适合用于加载配置文件（如 .bashrc、环境初始化脚本）

🔹 方式 4：使用 . 点命令

💡 . 与 source 功能完全等价。. 更简洁，source 语义更直观，按团队规范选用即可。

二、核心差异：子 Shell vs 当前 Shell

执行方式 运行环境 环境变量/变量变更影响 典型场景
bash/sh script.sh 🌲 子 Shell ❌ 不影响父 Shell 临时任务、隔离执行
./script.sh 🌲 子 Shell ❌ 不影响父 Shell 常规脚本调用
source script.sh 💻 当前 Shell ✅ 直接影响当前会话 加载配置、定义全局函数
. script.sh 💻 当前 Shell ✅ 直接影响当前会话 同 source，更简短

📌 关键结论：
方式 1 和 2 会在子 Shell 中运行，脚本内部的变量修改、cd 目录切换等操作不会污染或改变父 Shell 的环境。若希望脚本执行后终端环境随之改变（如激活虚拟环境、加载别名），必须使用 source 或 .。

三、系统预定义变量与查看命令

🔸 常用系统变量

🔸 局部变量 vs 全局变量

  • 局部变量：仅在当前 Shell 或特定子 Shell 中有效，脚本结束即销毁。
  • 全局变量：通过 export 导出，可传递给子进程/子 Shell 使用。
  • ⚠️ 重要机制：在子 Shell 中对变量的更改不会反向影响父 Shell。这也是为什么 bash script.sh 和 ./script.sh 执行后，无法在父终端中读取到脚本内定义的局部变量。

🔸 查看变量的三种命令

命令 作用范围 特点
env 仅全局环境变量 输出干净，适合脚本解析
printenv 仅全局环境变量 支持 printenv 变量名 精准查看
set 全部变量（全局+局部+函数） 信息最全，包含 Shell 内部状态

五、特殊变量速查表 🚀

在编写带参数的脚本时，这些内置变量是必须掌握的核心：

变量 含义 使用示例
$0 脚本名称/路径 echo "当前执行: $0"
$1 ~ $9 第 1~9 个位置参数 ./deploy.sh $1 $2
${10} 第 10 个及以上参数 必须加大括号 ${10}
$# 传入参数的总个数 if [ $# -eq 0 ]; then echo "请传参"; fi
$* 所有参数（视为一个整体字符串） 遍历输出时只循环 1 次
$@ 所有参数（视为独立字符串列表） 遍历输出时按原始参数个数循环 ✅ 推荐
$? 上一条命令的退出状态码 0 表示成功，非 0 表示失败

💡 笔记总结

  1. 执行选环境：需要隔离用 ./ 或 bash，需要生效到当前终端用 source 或 .。
  2. 变量守规范：赋值等号无空格，引用建议加 ${}，全局传递靠 export。
  3. 参数重边界：遍历命令行参数时，永远优先使用 "$@" 而非 "$*"。
  4. 子 Shell 隔离：子进程修改变量不会影响父 Shell，这是 Linux 进程模型的安全设计。
